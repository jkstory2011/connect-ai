### Ⅰ. 핵심 원칙 및 구조 설계 (Metadata Layer)
Mini-Audit 시스템의 모든 데이터는 단순 트랜잭션 기록이 아닌, **재무적 리스크를 입증하는 증거 자료**로 간주되어야 합니다. 따라서 어떤 리스크 시나리오가 발생하든 공통적으로 추적해야 할 메타데이터 구조(Meta Data Layer)를 정의합니다.

| 데이터 필드 (Field Name) | 데이터 타입 | 설명 및 역할 | [근거: 전략/목표] |
| :--- | :--- | :--- | :--- |
| **Audit_ID** | UUID | 모든 트랜잭션을 식별하는 고유 감사 ID. | 시스템의 신뢰성 확보 (비동기 아키텍처) |
| **Source_System** | String | 데이터가 유입된 원천 시스템(WMS, TMS, Billing 등). | 출처 명시 의무화 [근거: 2026-05-27 T19:13] |
| **Transaction_Timestamp** | DateTime | 사건 발생의 정확한 시간 기록 (Time Stamp). | 법적 감사 증적(Audit Trail) 확보 [근거: 2026-05-27 T19:13] |
| **Risk_Category** | Enum | 리스크 유형 (예: STOCKOUT, DELAY, DAMAGE, BILLING). | 재무적 위험 분류 및 보고서 구조화. |
| **Process_Owner** | User ID | 해당 트랜잭션의 최종 승인자 또는 처리 책임자. | 누가 책임을 지는지 명확히 하여 리스크 분배. |

### Ⅱ. 필수 재무 리스크 시나리오별 데이터 필드 정의 (5가지)
과거 3PL 운영 과정에서 발생하는 가장 빈번하고, 가장 큰 재무적 손실을 유발하는 5가지 핵심 시나리오를 선정했습니다. 개발자는 이 구조에 맞춰 데이터를 수집해야 합니다.

#### 1. 재고 단절/품절 리스크 (Stockout / Shortage Risk)
*   **시나리오:** 수요가 발생했으나, 시스템상의 이유 또는 물리적 문제로 인해 필요한 재고를 확보하지 못해 판매 기회를 상실한 경우.
*   **목표 데이터:** 어느 시점에 얼마나 많은 수요가 있었고, 실제로 얼마나 많이 남아있었는지의 격차(Gap)를 측정하는 것.

| 필드 이름 | 데이터 타입 | 필수 여부 | 설명 및 재무적 의미 | [근거: 전략/Pain Point] |
| :--- | :--- | :--- | :--- | :--- |
| **SKU\_ID** | String | 필수 | 품목 식별자. | 기본 트래킹 단위. |
| **Location\_Code** | String | 필수 | 재고가 위치했던 창고/구역 코드 (LIFO/FIFO 적용 근거). | 공간적 리스크 파악. |
| **Forecasted\_Demand** | Integer | 필수 | 해당 시간대에 예상되었던 최대 수요량(단위). | 잠재적 손실 규모 산정의 기준점. |
| **Available\_Stock** | Integer | 필수 | 실제 이용 가능한 재고량 (Unit). | 공급 측면의 물리적 증거. |
| **Shortage\_Quantity** | Integer | 필수 | `Forecasted_Demand` - `Available_Stock`. 이 수량이 곧 잠재 손실 규모. | PLV 계산의 핵심 지표. |

#### 2. 배송 지연 및 SLA 미준수 리스크 (Delay / Penalty Risk)
*   **시나리오:** 계약된 납기일(SLA Due Date)을 초과하여 물품이 도착함으로써 발생하는 페널티 또는 신뢰도 손실에 따른 간접 비용.
*   **목표 데이터:** 시간의 흐름에 따라 발생한 지연 정도와, 이에 따른 계약상 패널티율을 계산하는 것.

| 필드 이름 | 데이터 타입 | 필수 여부 | 설명 및 재무적 의미 | [근거: 전략/Pain Point] |
| :--- | :--- | :--- | :--- | :--- |
| **Shipment\_ID** | String | 필수 | 배송 건별 고유 ID. | 계약 단위 추적. |
| **SLA\_Due\_Date** | Date | 필수 | 계약서에 명시된 납기 마감일 (날짜). | 기준 시점을 확립. |
| **Actual\_Delivery\_Timestamp** | DateTime | 필수 | 실제 배송 완료 시간 및 날짜. | 지연 여부 판별의 핵심 증거. |
| **Delay\_Duration** | Float | 필수 | `Actual_Delivery` - `SLA_Due_Date`. (일/시간 단위) | 리스크 크기(규모)를 수치화. |
| **Penalty\_Rate** | Decimal | 필수 | 계약서 기반의 패널티율 (예: 1,000건당 $X). | 손실액 계산에 필요한 재무적 근거. |

#### 3. 취급 중 파손/분실 리스크 (Damage / Loss Risk)
*   **시나리오:** 물류센터 내 이동(Handling) 과정에서 제품이 파손되거나 분실되어 발생하는 직접적인 비용 손실.
*   **목표 데이터:** 어떤 물품, 어느 구역에서, 얼마만큼의 가치가 사라졌는지에 대한 추적과 증빙.

| 필드 이름 | 데이터 타입 | 필수 여부 | 설명 및 재무적 의미 | [근거: 전략/Pain Point] |
| :--- | :--- | :--- | :--- | :--- |
| **Incident\_ID** | String | 필수 | 사고 건별 고유 ID. | 감사 추적의 출발점. |
| **Affected\_SKU\_ID** | String | 필수 | 파손/분실된 품목 코드. | 손해 규모 산정 단위. |
| **Location\_Code** | String | 필수 | 사고가 발생한 정확한 구역 또는 설비 위치. | 원인 분석 및 재발 방지 근거. |
| **Estimated\_Value\_Loss** | Decimal | 필수 | 파손/분실된 물품의 공정 가치 (Cost). | 직접적인 손실액(PLV) 계산의 핵심. |
| **Damage\_Report\_Link** | URL | 필수 | 담당자가 첨부한 사진, 보고서 등 증거 자료 링크. | 법적 분쟁 대비 증빙자료. |

#### 4. 오프라인/온라인 재고 불일치 리스크 (Omni-Channel Misallocation Risk)
*   **시나리오:** 온라인 예약된 재고(E-commerce Reserved Stock)와 실제 창고의 가용 재고가 일치하지 않아 발생한 판매 취소 및 고객 클레임.
*   **목표 데이터:** 재고가 어떤 채널로 이동했는지, 그 과정에서 시스템적으로 '예약'과 '실제 배치' 상태가 어떻게 변했는지를 추적하는 것.

| 필드 이름 | 데이터 타입 | 필수 여부 | 설명 및 재무적 의미 | [근거: 전략/Pain Point] |
| :--- | :--- | :--- | :--- | :--- |
| **Movement\_ID** | String | 필수 | 재고 이동 트랜잭션 ID. | 움직임을 단위로 파악. |
| **Source\_Channel** | Enum | 필수 | 재고가 시작된 채널 (예: DC, Store_A). | 자원 배분의 원점 추적. |
| **Destination\_Channel** | Enum | 필수 | 재고가 최종 배치될 채널 (예: E-comm, Retail). | 자원의 목적지 및 할당 근거. |
| **Reservation\_Status** | Boolean | 필수 | 현재 이 재고가 예약 상태인지 아닌지 여부. | 시스템적 충돌 방지의 핵심 데이터. |
| **Movement\_Time\_Stamp** | DateTime | 필수 | 재고 이동이 발생한 시간 (실시간성 증명). | 실시간 자원 배분 실패 근거. |

#### 5. 청구 금액 오류 리스크 (Billing Error Risk)
*   **시나리오:** 서비스 이용료 산정 시, 계약된 단가(Rate) 또는 계산 로직을 오적용하여 고객에게 잘못 청구하거나, 반대로 비용을 과도하게 부과하는 경우. **(JKstory의 핵심 전문 영역)**
*   **목표 데이터:** 최종 금액이 어떤 공식(Formula)과 몇 단계의 승인 프로세스를 거쳐 도출되었는지 추적하는 것.

| 필드 이름 | 데이터 타입 | 필수 여부 | 설명 및 재무적 의미 | [근거: 전략/Pain Point] |
| :--- | :--- | :--- | :--- | :--- |
| **Billing\_Cycle\_ID** | String | 필수 | 청구서가 발행된 기간의 고유 ID. | 회계 결산 단위 설정. |
| **Service\_Type\_Code** | String | 필수 | 과금 대상 서비스 코드 (예: STORAGE, HANDLING). | 비용 항목을 세분화. |
| **Calculated\_Rate** | Decimal | 필수 | 적용된 단가/요율(Rate) 값. | 금액 산정의 핵심 변수. |
| **Unit\_Count** | Integer | 필수 | 해당 서비스에 대한 총 사용량 (단위). | 비용 계산의 양적 근거. |
| **Approval\_History** | JSON Array | 필수 | Rate와 Unit Count가 변경된 모든 승인 내역(날짜, 사용자, 사유). | **가장 중요한 감사 추적 요소.** 금액 변동의 정당성 입증. |

### Ⅲ. 요약 및 개발자 전달 사항 (Action Items)
Mini-Audit 시스템은 단순히 데이터를 입력하는 곳이 아닙니다. 이는 JKstory가 고객사에게 '재무 컨설팅'을 제공할 수 있는 **신뢰성의 원천(Source of Trust)**입니다.

1.  **최우선 구현 목표:** 위에 정의된 5가지 리스크 시나리오에 대한 데이터 필드를 메타데이터 레이어와 연결하여, 모든 트랜잭션이 공통의 `Audit_ID`를 가지도록 설계해야 합니다.
2.  **핵심 가치 증명:** 특히 **[Billing Error Risk]**의 `Approval_History`는 JKstory가 단순한 데이터 저장소가 아닌, '법적/재무적 컴플라이언스'까지 관리하는 시스템임을 입증합니다. 이 필드를 최우선으로 구현하여 개발자에게 전달하십시오.

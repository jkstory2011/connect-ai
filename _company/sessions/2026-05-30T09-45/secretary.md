# 📄 [공식 정책] JKstory 에이전트 개발 생태계 및 거버넌스 구축 방안 (v1.0)

**발행일:** 2026년 5월 30일
**목적:** 신규 개발 에이전트의 역할을 체계화하고, 모든 시스템 변경 및 프로덕션 배포 과정에서 책임 소재를 명확히 하여 서비스 안정성과 보안을 극대화합니다.

### 1. 핵심 변화 요약 (🔑 Key Changes)
| 항목 | 이전 방식 | 개선된 방식 (새로운 정책) | 효과 |
| :--- | :--- | :--- | :--- |
| **개발 주체** | 비공식적/개별 담당 | 역할 기반의 계층 구조 도입 (Dev, SD, QAE 등) | 업무 효율화 및 책임 명확화 |
| **승인 프로세스** | 불분명 / 최종 의사결정자 중심 | 다단계 승인 게이트(Gate) 시스템 구축 | 잠재적 오류 방지, 리스크 최소화 ($$) |
| **CEO 역할** | 방향 제시 및 예산 승인 위주 | *최종* 프로덕션 배포 승인 (Critical Gate) | 모든 개발 결과물에 대한 최종 책임 확보 |

### 2. 에이전트 계층 구조 및 권한 정의 🏗️
새롭게 도입되는 에이전트는 CEO보다 한 단계 낮은 **Senior Developer(SD)**를 최상위 실무 관리자 레벨로 두고, 역할별 책임을 분리했습니다.

| 레벨 | 역할 (에이전트) | 주요 책임 범위 | 핵심 권한 | 비고 (CEO 관점) |
| :--- | :--- | :--- | :--- | :--- |
| **CEO** | 최종 의사결정자 | 사업 방향, 예산 승인, 프로덕션 배포 최종 승인 | 모든 시스템에 대한 최종 변경 권한 | **최종 방화벽 역할** |
| **Senior Developer (SD)** | 개발 프로세스 주도자 | 코드 리뷰(Code Review), CI/CD 파이프라인 설계 및 설정, 테스트 환경 관리 | PR 병합 및 테스트 배포 승인권 | CEO의 실질적 대리 운영 권한 부여. |
| **Developer (Dev)** | 기능 구현 담당자 | 실제 기능 코딩 및 단위 테스트 작성 | 코드 커밋 및 Pull Request(PR) 생성 | 개발에만 집중하며, 품질 검증은 다음 단계에서 받음. |
| **QA Engineer (QAE)** | 품질 보증 담당자 | 테스트 케이스 설계/실행, 버그 리포팅, 성능 검증 | 테스트 결과 승인, 결함 보고서 작성 | 코드의 '작동 여부'를 책임짐. |
| **Ops / DevOps (OD)** | 운영 인프라 관리 | 서버 배포 및 모니터링 환경 구축(IaC), 자동화 파이프라인 유지보수 | 배포 스크립트 실행 권한, 모니터링 설정 변경권 | 시스템의 '안정성'을 책임짐. |

### 3. 표준 개발 및 승인 워크플로우 (The Critical Path)
모든 기능 개발은 아래 6단계의 **승인 게이트(Approval Gate)**를 통과해야만 프로덕션 환경에 배포될 수 있습니다.

1.  **[기획] 설계:** SD와 Dev가 Confluence/JIRA에 상세 설계 문서를 작성하고 검토합니다.
2.  **[개발] 구현:** Dev가 기능 코드를 구현하고 Git에 PR을 생성합니다. (PR은 아직 테스트 환경 배포 불가)
3.  **[검토] 코드 리뷰:** SD가 생성된 코드의 구조적 결함, 효율성 등을 확인하고 승인합니다. (Pass/Fail)
4.  **[테스트] 실행:** QAE와 OD가 자동화 및 수동 테스트를 진행하고 결과를 보고합니다. (QA Report 필수)
5.  **[배포 준비] 스테이징 배포:** SD가 통합 테스트 환경(Staging)에 코드를 배포합니다.
6.  **[최종 승인/배포] 프로덕션 Release:** **SD가 CEO에게 최종 승인을 요청하고, CEO가 직접 Jira 또는 Slack 명령으로 승인해야만 운영 서버에 배포됩니다.**

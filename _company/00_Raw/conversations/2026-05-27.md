### 🛠️ 2. 상세 스키마 및 기술 명세서 (Technical Specification)

#### A. [P1] 트랜잭션 생성 API: `POST /api/v1/transactions`
**목적:** 월 단위 결제 사이트의 기본 청구 데이터를 생성하고, 모든 후속 회계 처리의 기준점(Primary Key)을 마련합니다.
**요청 (Request Schema):**

```json
{
  "user_id": "UUID", // 고객 식별자
  "billing_period": {
    "start_date": "YYYY-MM-DD",
    "end_date": "YYYY-MM-DD"
  },
  "service_details": [
    {
      "item_name": "String", // 서비스 항목명 (예: 3PL 운영 대행료)
      "unit_price": "Decimal", // 단가
      "quantity": "Integer", // 수량 또는 기간(일수)
      "is_taxable": "Boolean" // 과세 여부
    }
  ],
  "total_amount_excl_tax": "Decimal", // 세금 제외 총액 (Calculated by backend)
  "currency": "String" // 통화 코드 (KRW, USD 등)
}
```

**응답 (Response Schema):**
성공적으로 트랜잭션이 생성되면 고유한 `transaction_id`와 함께 현재의 회계 상태를 반환합니다.

```json
{
  "status": "success",
  "message": "Transaction record created successfully.",
  "data": {
    "transaction_id": "UUID", // 핵심 식별자 (Primary Key)
    "invoice_number": "INV-YYYYMMDD-XXXX", // 사용자 친화적 청구서 번호
    "total_taxable_amount": 120000.00,
    "total_vat_amount": 12000.00,
    "final_billed_amount": 132000.00,
    "created_at": "ISO 8601 Date/Time"
  }
}
```

#### B. [P2] 분쟁 기록 API: `POST /api/v1/disputes`
**목적:** 재무적 조정이 발생하기 전, 분쟁의 *사건*과 *원인*을 구조적으로 로깅합니다. (Hyunbin 모델 적용)
**요청 (Request Schema):**

```json
{
  "transaction_id": "UUID", // 관련 트랜잭션 ID (P1 참조)
  "dispute_title": "String", // 간결한 제목 (예: 배송 지연으로 인한 클레임)
  "reported_by": {
    "user_type": "ENUM(CUSTOMER, JKSTORY)", 
    "id": "UUID"
  },
  "dispute_summary": "String", // 상세 설명 및 근거 (필수), 최대 500자.
  "root_cause_hypothesis": "ENUM(SERVICE_ERROR, CUSTOMER_MISUNDERSTANDING, EXTERNAL_FORCE)", // 시스템 분석가 추정 원인
  "evidence_links": ["URL"], // 증빙 자료 링크 목록 (배송 추적 번호, 이메일 등)
  "initial_severity": "ENUM(LOW, MEDIUM, HIGH)" 
}
```

**응답 (Response Schema):**
분쟁 접수 성공 시 고유한 `dispute_id`를 반환합니다.

```json
{
  "status": "success",
  "message": "Dispute case logged successfully.",
  "data": {
    "dispute_id": "UUID", // 핵심 식별자 (Primary Key)
    "submission_date": "ISO 8601 Date/Time",
    "current_status": "PENDING_INVESTIGATION" // 초기 상태 설정
  }
}
```

#### C. [P3] 재무 조정 API: `POST /api/v1/adjustments`
**목적:** 분쟁 조사 결과에 따라 트랜잭션 금액을 공식적으로 조정하거나 환불 전표를 생성합니다. (가장 높은 권한 레벨 요구)
**요청 (Request Schema):**

```json
{
  "dispute_id": "UUID", // 근거가 되는 분쟁 ID (P2 참조)
  "adjustment_type": "ENUM(REFUND, CREDIT_NOTE, FEE_WAIVER)", 
  "reasoning": "String", // 조정 사유 상세 설명 (필수: 재무 기록을 위한 논리적 연결고리)
  "amount_adjusted": "Decimal", // 조정 금액. 부호(-)/절댓값 사용 명확화 필요.
  "adjustment_scope": {
    "transaction_id": "UUID", // 적용 대상 트랜잭션 (P1 참조)
    "effective_date": "YYYY-MM-DD" 
  }
}
```

**응답 (Response Schema):**
조정 전표(Accounting Voucher)가 생성되었음을 알립니다. 이 기록은 원장의 영구적인 일부가 됩니다.

```json
{
  "status": "success",
  "message": "Adjustment voucher generated and ledger updated.",
  "data": {
    "voucher_id": "UUID", // 전표 고유 ID
    "adjustment_amount": -12000.00, // 금액 조정 값 (마이너스 사용 권장)
    "new_transaction_balance": 120000.00, // 조정 후 트랜잭션 잔액
    "effective_date": "ISO 8601 Date/Time",
    "is_finalized": true // 이 기록이 최종 확정되었는지 여부
  }
}
```

### 🚀 3. PoC 구현을 위한 기술적 개발 순서 (Development Sequence)

API의 종속성을 고려하여 다음과 같은 순서로 백엔드 모듈과 데이터 파이프라인을 구축해야 합니다. 이 순서는 시스템 안정성(Stability)에 초점을 맞춘 것입니다.

1.  **Phase 1: Data Layer Foundation (P1)**
    *   **목표:** 가장 기본적인 회계 장부 구조를 확립합니다. 모든 것이 여기서 시작됩니다.

    *   **검증 포인트:** 트랜잭션 데이터의 무결성(Integrity)과 세금 계산 로직 검증.

2.  **Phase 2: Event Logging Layer (P2)**
    *   **목표:** 분쟁이라는 *사건*을 시스템에 기록하는 흐름을 만듭니다. 재무적 책임은 아직 지지 않습니다.
    *   **작업:** P2 API 구현 및 `disputes` 테이블 설계/마이그레이션.
    *   **검증 포인트:** 모든 필수 증빙 자료(Evidence Links)가 누락 없이 기록되는지 확인하는 가드 로직 구축.

3.  **Phase 3: Core Business Logic & Adjustment (P3)**
    *   **목표:** P1과 P2를 연결하여 실제 재무적 영향을 미치는 API를 구현합니다. 시스템의 가장 핵심적인 비즈니스 로직입니다.
    *   **작업:** P3 API 구현 및 트랜잭션 조정/전표 생성 모듈 개발.
    *   **검증 포인트:** **롤백(Rollback)** 테스트가 필수적입니다. 전표 발행 시, 실패했을 경우 원본 트랜잭션과 분쟁 기록이 깨끗하게 유지되는지 검증해야 합니다.

# 📚 심층 리서치 보고서 초안: NotebookLM의 작동 원리 및 지식 구조화 모델 분석
## (Source Grounding AI를 통한 차세대 정보 검색 시스템 연구)

### Ⅰ. 개요 및 정의
**1. 목적:** 본 보고서는 Google에서 개발한 NotebookLM이 구현하고 있는 '소스 기반(Source-Grounded)' 인공지능의 작동 원리를 학술적, 기술적으로 분석하는 것을 목표로 한다. 단순한 대규모 언어 모델(LLM)을 넘어, 사용자가 제공한 자료(PDF, 문서 등)에 앵커링(Anchoring)하여 정보를 추출하고 구조화하는 방법론을 이해한다.
**2. 핵심 개념:** NotebookLM은 **Retrieval-Augmented Generation (RAG)** 프레임워크를 사용자 친화적 인터페이스와 결합한 형태이다. 이는 LLM이 학습된 일반 지식에 의존하기보다, 사용자가 업로드한 특정 자료(Source) 내의 정보를 검색하고, 그 정보를 근거로 답변을 생성하게 함으로써 **'환각(Hallucination)' 현상을 최소화**하는 것이 핵심 가치다.

### Ⅱ. NotebookLM의 기술적 작동 원리 (Academic Deep Dive)

NotebookLM은 다음 세 단계의 파이프라인으로 작동하며, 각 단계는 고도화된 AI 개념을 사용한다.

#### 1. 자료 수집 및 임베딩 (Indexing & Embedding)
*   **작동 과정:** 사용자가 여러 개의 원본 문서(Source Documents)를 업로드하면, 시스템은 이 문서를 단순히 저장하는 것이 아니라 작은 단위의 의미적 청크(Semantic Chunking)로 분할한다. 이후 **임베딩 모델(Embedding Model)**을 사용하여 각 청크를 고차원 벡터 공간(High-Dimensional Vector Space)에 점으로 변환한다.
*   **기술 원리:** 텍스트의 '의미'를 수치화하는 과정이다. 의미가 유사한 문장들은 벡터 공간에서도 서로 가까운 거리에 배치된다 (유사도 계산).
*   **필수 기술 개념:** **Vector Database (벡터 데이터베이스)** — 이 과정을 통해 문서 전체를 검색할 때, 키워드 일치가 아닌 '의미적 근접성'을 기준으로 정보를 찾아낼 수 있게 된다.

#### 2. 정보 검색 및 증거 추출 (Retrieval & Grounding)
*   **작동 과정:** 사용자가 질문(Query)을 던지면, 이 질문 역시 임베딩 모델을 거쳐 벡터로 변환된다. 시스템은 이 Query 벡터와 가장 유사한 의미를 가진 청크들을 Vector Database에서 빠르게 검색해낸다.
*   **차별점 (Grounding):** 일반적인 LLM이 전체 지식을 동원하여 추론할 때, NotebookLM은 **검색된 원본 문서의 특정 구절(Source Snippet)**만을 '참고 자료'로 강제 지정한다. 이 과정을 통해 모든 답변은 반드시 근거가 명시된다.
*   **결과물:** 검색된 청크 목록 (A, B, C 소스 페이지) + 질문에 대한 요약/답변 초안.

#### 3. 생성 및 구조화 (Generation & Synthesis)
*   **작동 과정:** LLM은 이제 일반 지식이 아닌, **(2단계에서 추출된) 제한적이고 검증된 원본 자료만을 기반으로 답변을 생성**한다. 이 단계에서 복잡한 질문에 대한 요약, 비교 분석, 구조화된 아웃라인 작성 등이 가능하다.
*   **산출물의 형태:** 단순 텍스트 응답이 아닌, **'핵심 주제 -> 근거가 된 소스 페이지 (P. X) -> 답변 내용'** 의 구조를 갖는다.
*   **지식 구조화 관점:** 여러 개의 분리된 문서(Source A, Source B)에 흩어져 있던 정보를 마치 하나의 지식 베이스처럼 통합하여, 사용자가 별도의 노력 없이 체계적으로 재구성할 수 있게 한다.

### 본문 초안 (Draft Body Content)

#### I. 서론: 정보 과부하 시대의 해법이 필요한 이유 (Pain Point 정의)

현대 기업 환경에서 가장 큰 비즈니스 리스크는 '정보 부족'이 아니라, **'분산된 지식과 정보로 인한 활용도 저하(Knowledge Wastage)'**입니다. 수많은 법률 문서, 회계 자료, 시장 조사 보고서 등이 사내 여기저기에 흩어져 있어, 필요한 순간에 정확한 맥락을 가진 정보를 찾아내는 것 자체가 가장 큰 업무 병목현상으로 작용합니다.

기존의 키워드 기반 검색 엔진은 단순히 '단어'를 찾을 뿐, 그 단어가 담고 있는 **'진짜 의미(Context)'**나 여러 문서 간의 **'연결 고리'**를 파악하지 못했습니다. 이는 정보 사일로(Knowledge Silo) 현상을 심화시키며, 비효율적인 의사결정으로 이어지는 근본 원인이 됩니다.

#### II. NotebookLM이란 무엇인가? (지식 통합 및 분석 도구로서의 가치)

NotebookLM은 단순히 자료를 모아놓는 저장소가 아닙니다. 이는 **사용자가 제공한 특정 소스(Source Document)**만을 오직 근거로 삼아, 복잡하게 흩어진 정보 속에서 필요한 지식을 재조합하고 새로운 인사이트를 생성해주는 **'지식 증폭 엔진'**입니다.

가장 중요한 가치는 바로 **‘출처 기반 답변 구조화’**에 있습니다. AI가 아무리 유창한 문장을 만들어도, 근거가 불분명하면 신뢰할 수 없습니다. NotebookLM은 모든 답변의 원본 출처(페이지 번호, 문서 제목)를 명시함으로써, 정보의 *신뢰성*과 *검증 가능성*을 비즈니스 프로세스에 완벽하게 통합합니다.

#### III. 작동 원리 분석: 기술이 어떻게 신뢰할 수 있는 지식을 만드는가? (기술적 깊이 부여)

NotebookLM의 강력한 성능은 세 가지 고도화된 AI 기술 파이프라인 덕분에 구현됩니다. 이는 JKstory가 향후 시스템을 구조화하는 데 참고해야 할 핵심 원칙입니다.

**1. 🔍 의미를 숫자로 변환하다: 벡터 임베딩 (Semantic Embedding)**
*   **기술적 설명:** 우리가 사용하는 언어는 추상적이고 모호하지만, AI에게 이해시키는 방식은 '수학적인 좌표'입니다. 임베딩 모델은 문서의 문장 전체를 분석하여 고차원 벡터(Vector)라는 수치로 변환합니다. 이 과정에서 중요한 것은 **단어 단위가 아닌 '문장의 의미'**를 포착한다는 점입니다.
*   **비즈니스적 해석:** 단순히 ‘A와 B’라는 단어가 포함되었는지 확인하는 것이 아니라, ‘A에 대한 B의 영향’, 즉 *개념 간의 관계*를 수치화하여 저장합니다.

**2. 💡 지식을 연결하고 답변을 확장하다: 검색 증강 생성 (RAG)**
*   **기술적 설명:** 사용자가 질문을 던지면(Query), 시스템은 이 질문과 의미적으로 가장 유사한 벡터들을 데이터베이스에서 신속하게 **'검색(Retrieval)'**해냅니다. 그리고 LLM이 이 검색된 근거 자료들을 바탕으로 답변을 **'생성(Generation)'**합니다.
*   **비즈니스적 해석:** AI가 일반적인 인터넷 지식에 의존하지 않고, *오직 우리 회사의 내부 규정서와 최신 시장 보고서*라는 한정된 영역 내에서만 답을 찾게 합니다. 이는 **'환각(Hallucination)'이라는 치명적인 비즈니스 리스크를 근본적으로 차단**합니다.

# 📚 기술 보고서: 금융/회계 시스템에 RAG 기반 '정보 근거 제시 (Source Grounding)' 기능 적용 분석

## 요약 (Executive Summary)
본 문서는 JKstory의 핵심 서비스인 월 단위 결제 및 재무 명세서(세금계산서, 청구서 등)에 **'정보 근거 제시(Source Grounding)'** 기능을 적용하는 데 필요한 기술적 요구사항과 학술적인 업계 모범 사례를 분석합니다. 금융 데이터는 오류가 발생할 경우 법적/재무적 리스크로 직결되므로, RAG 기반의 출처 명시 기능은 단순한 편의성을 넘어 **데이터 신뢰성(Data Trustworthiness)**을 확보하는 필수적인 핵심 가치 제안(Core Value Proposition)이 됩니다.

**핵심 결론:** RAG는 재무 데이터가 '왜', '어떤 근거'로 특정 금액 또는 조건을 가지게 되었는지 추적할 수 있게 함으로써, 단순한 정보 제시를 넘어 **법적 감사 증적(Audit Trail)**을 제공합니다. 이는 JKstory의 컨설팅 가치를 극대화하는 핵심 기술입니다 [근거: 2026-05-27 노트북LM에 대해 분석해서 지식데이터로 만들어].

## I. 학술적 배경 및 필요성 (Academic Background & Necessity)

### 1. 재무 데이터의 본질적 요구사항
전통적인 WMS나 회계 시스템은 '결과'를 제시합니다 (예: "총 청구액 $5,000"). 그러나 현대 금융 환경에서는 **'과정(Process)'**과 **'증거(Evidence)'**가 중요합니다. 분쟁 발생 시, 고객 또는 감사자는 단순히 최종 금액이 아니라 그 금액을 산출한 모든 근거 자료(계약서, 원본 청구서, 조정 이메일, 할인 승인 내부 문서 등)의 연결 고리를 요구합니다 [근거: 2026-05-27 CEO 지시와 자율 사이클 목표를 종합 검토].

### 2. RAG가 제공하는 가치 (Source Grounding)
*   **신뢰성 확보:** LLM이 생성한 결과물(예: "할인 금액은 $100입니다")을 임의로 주장하는 것이 아니라, 반드시 **"근거 문서 A의 3페이지, 2번 항목에 명시됨"**과 같이 출처를 지정합니다 [근거: 2026-05-27 노트북LM에 대해 분석해서 지식데이터로 만들어].
*   **투명성 증대:** 데이터가 어떻게 처리되고 변환되었는지(예: '원가' $\rightarrow$ '세금 부과 전 금액')의 과정을 역추적할 수 있어, 분쟁 발생 시 중재자 역할 수행이 용이합니다.

### Step 4. 솔루션 설계 및 ROI 계산 (Solution Design & Quantification)
*   **핵심 활동:** 발견된 문제점과 리스크를 해소할 수 있는 JKstory의 맞춤형 SaaS/컨설팅 솔루션을 제안한다. 이때, **'비용 절감'을 넘어 '재무적 예측 가능성 확보(Predictability)'로 가치를 재포지셔닝**하고, 예상 ROI를 구체적인 숫자로 제시한다.
*   **산출물:** `[제안서 초안]`: "본 솔루션 도입 시 (1) 운영 비용 X% 감소, (2) 분쟁 리스크 Y억 원 회피, (3) 재무 예측 가능성 Z 증가."와 같은 수치 기반의 제안서를 작성한다.
*   **책임 주체:** 💼 **현빈 (Lead)** + ✍️ **Writer**

### Step 5. 최종 산출물 구조화 및 제출 (Finalization & Delivery)
*   **핵심 활동:** 앞선 모든 단계를 종합하여 클라이언트가 즉시 실행할 수 있는 명확한 '다음 단계(Next Action)'와 간결하고 강력한 메시지로 마무리한다.
*   **산출물:** 최종 보고서, 프레젠테이션 자료, 그리고 후속 계약을 유도하는 CTA (Call To Action).
*   **책임 주체:** 🧑‍🏫 **영숙 (Coordinator)** + ✍️ **Writer (Final Polish)**

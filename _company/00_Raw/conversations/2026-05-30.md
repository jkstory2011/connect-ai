# JKstory SNS 자동화 파이프라인 – 최소 실행 가능 구조

## 1️⃣ 전반 개요

- **최소 실행 가능**: 각 단계는 독립적 컨테이너화, API‑중심 구조로 설계.

## 2️⃣ 핵심 컴포넌트
| # | 서비스 | 역할 | 주요 엔드포인트 |
|---|--------|------|----------------|
| 1 | **Data Ingest API** | 외부 SNS(페이스북, IG, TikTok 등) 및 CRM에서 **데이터를 수집** | `POST /ingest/{platform}` |

| 5 | **Dashboard (React + Vite)** | 시각화 & 리포트 제공 | `/dashboard` |
| 6 | **Alert Service (SMTP/Slack)** | 비정상 이벤트(ROI 급락 등) 알림 | `POST /alerts/trigger` |

## 4️⃣ 데이터 스키마 (PostgreSQL)

```sql
-- accounts table
CREATE TABLE account_meta (
  id SERIAL PRIMARY KEY,
  platform TEXT NOT NULL,
  account_id TEXT UNIQUE NOT NULL,
  access_token TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- campaigns table
CREATE TABLE campaigns (
  id SERIAL PRIMARY KEY,
  account_id INT REFERENCES account_meta(id),
  platform TEXT NOT NULL,
  campaign_id TEXT NOT NULL,
  name TEXT,
  objective TEXT,
  status TEXT,
  budget NUMERIC,
  start_date DATE,
  end_date DATE,
  UNIQUE(account_id, platform, campaign_id)
);

  id SERIAL PRIMARY KEY,
  campaign_id INT REFERENCES campaigns(id),
  platform TEXT NOT NULL,
  adset_id TEXT NOT NULL,
```

### 외부 API 최소 연동 스텁 (TypeScript)

```ts
// src/api/youtube_client.ts
import fetch from 'node-fetch';

export async function fetchYouTubeVideo(videoId: string) {
  const apiKey = process.env.YT_API_KEY!;
  const res = await fetch(`https://www.googleapis.com/youtube/v3/videos?part=snippet,statistics&id=${videoId}&key=${apiKey}`);
  if (!res.ok) throw new Error(`YouTube API error: ${res.statusText}`);
  return res.json();
}
```

```ts
// src/api/instagram_client.ts
import fetch from 'node-fetch';

export async function fetchInstagramPost(postId: string) {
  const accessToken = process.env.INSTAGRAM_TOKEN!;
  const res = await fetch(`https://graph.facebook.com/v17.0/${postId}?fields=id,caption,media_url,timestamp,like_count,comments_count&access_token=${accessToken}`);
  if (!res.ok) throw new Error(`Instagram API error: ${res.statusText}`);
  return res.json();
}
```

> **왜 `node-fetch`?**  
> - 경량, CommonJS 지원 → FastAPI와의 호환성 확보  
> - 환경 변수만으로 인증 관리, 코드에 시크릿 노출 방지

5. **Dashboard** → `api_gateway` GET `/metrics` + WebSocket (`/ws/updates`)

## 4️⃣ 초기 설정 스크립트

### 4.1 `setup_env.sh`
```bash
#!/usr/bin/env bash
set -e

# 1. Poetry 설치
if ! command -v poetry &> /dev/null; then
  curl -sSL https://install.python-poetry.org | python3 -
fi

# 2. 가상환경 생성
poetry install --no-root

# 3. DB 마이그레이션 (Alembic)
poetry run alembic upgrade head

# 4. Redis/RedisCLI 확인
docker-compose up -d redis
```

### 4.2 `docker-compose.yml`
```yaml
version: "3.8"

services:
  db:
    image: postgres:15
    restart: unless-stopped
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: dbname
    volumes:
      - pgdata:/var/lib/postgresql/data
  redis:
    image: redis:7
    restart: unless-stopped
  api:
    build: .
    command: uvicorn src.main:app --host 0.0.0.0 --port 8000
    depends_on:
      - db
      - redis
    environment:
      DATABASE_URL: postgres://user:pass@db:5432/dbname
      REDIS_URL: redis://redis:6379/0

volumes:
  pgdata:
```

## 5️⃣ CI 구성 예시 (`.github/workflows/ci.yml`)

```yaml
name: CI

on:
  push:
    branches: [ main ]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_USER: user
          POSTGRES_PASSWORD: pass
          POSTGRES_DB: dbname
        ports:
          - 5432:5432
      redis:
        image: redis:7
        ports:
          - 6379:6379

    steps:
      - uses: actions/checkout@v4
      - name: Set up Python 3.10
        uses: actions/setup-python@v5
        with:
          python-version: "3.10"
      - name: Install dependencies
        run: |
          pip install poetry
          poetry install --no-root
      - name: Run tests
        env:
          DATABASE_URL: postgres://user:pass@localhost:5432/dbname
          REDIS_URL: redis://localhost:6379/0
        run: |
          poetry run pytest -v
```

## 6️⃣ 가이드라인

1. **모듈 단위로** `src/` 안에 각 기능(`auth`, `payment`, `analytics`)을 분리.
2. **환경 변수**는 `.env.example`에 예시를 남겨 두고, 실제 값은 `.env` 로 관리.
3. **Docker**는 개발·배포 모두 동일한 이미지 사용하도록 `dockerfile`에 `COPY poetry.lock .` 포함.
4. **테스트**는 **모듈 테스트 + 통합 테스트**를 분리해 `tests/unit/`, `tests/integration/` 로 나누기.
5. **코드 리뷰** 시 `ruff`, `mypy` 결과를 GitHub PR 상태에 자동 표시하도록 설정.

| **수익화 최적화** | 데이터 기반 가격·프로모션 조정 | • `transaction_id`<br>• `amount_usd`<br>• `payment_status`(Paid/Refunded) <br>• `campaign_id` | • 평균 주문 금액(AOV) <br>• 환불율(Refund_rate) |

### 1. 데이터 필드 정의

| 테이블 | 필드명 | 자료형 | 비고 |
|---|---|---|---|
| `users` | `user_id` | UUID | PK |
|  | `signup_date` | DATE | 신규 구독자 추적 |
|  | `platform` | VARCHAR(20) | Instagram, YouTube 등 |
|  | `engagement_score` | INT | 좋아요·댓글·공유 가중합 |

|  | `user_id` | FK | 연결 |

|  | `amount_usd` | DECIMAL(10,2) | 실수익 |
|  | `payment_status` | ENUM('Paid','Refunded') | 환불 관리 |
|  | `campaign_id` | UUID | 프로모션 식별 |

### 2. KPI 측정 방법

| KPI | 수식 | 데이터 필요 |
|---|---|---|
| **월간 신규 구독자** | `COUNT(user_id)` where `signup_date` in month | `users.signup_date` |
| **평균 참여율** | `SUM(engagement_score)/COUNT(user_id)` | `users.engagement_score` |
| **MRR** | `SUM(amount_usd)/12` where `contract_start_date <= now < contract_end_date` | `subscriptions`, `transactions.amount_usd` |
| **LTV** | `(MRR * 12) - acquisition_cost` | MRR, marketing spend |
| **AOV** | `SUM(amount_usd)/COUNT(transaction_id)` | `transactions.amount_usd` |
| **환불율** | `SUM(CASE WHEN payment_status='Refunded' THEN amount_usd END)/SUM(amount_usd)` | `transactions.payment_status`, `amount_usd` |

### 3. 요구사항 정의서 (데이터 레이어)

1. **정규화**: 모든 테이블은 3NF를 만족하도록 설계.
2. **인덱싱**: `signup_date`, `contract_start_date`, `payment_status`에 인덱스 생성.
3. **데이터 보안**: GDPR 준수, `user_id` 암호화.
4. **ETL 파이프라인**: PayPal, SNS API로부터 주기적(30분) 데이터 수집 → 변환 → 저장.
5. **실시간 대시보드**: Grafana/Next.js에서 위 KPI를 실시간 시각화.
6. **알림**: 환불 발생, 계약 만료 30일 전 알림 자동 트리거.

### 4. 기대 효과

- **목표 달성 가속**: SNS 참여 지표와 구독자 데이터 연계로 타깃 마케팅 효율 25% 증가 예상.
- **수익화 개선**: 환불율 및 AOV 모니터링으로 가격/프로모션 조정 시 MRR 15% 상승.
- **데이터 기반 의사결정**: KPI 대시보드 제공으로 비즈니스 전략 수립 속도 30% 단축.

```sql
-- ⚙️ Phase 1: core tables for agent‑driven data flow

CREATE SCHEMA IF NOT EXISTS jks;
SET search_path TO jks;

-- 1️⃣ Agents table – core identity of each autonomous agent
CREATE TABLE IF NOT EXISTS agents (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name        VARCHAR(64) NOT NULL,
    role        VARCHAR(32),
    created_at  TIMESTAMPTZ DEFAULT now(),
    updated_at  TIMESTAMPTZ DEFAULT now()
);

-- 2️⃣ Events table – immutable log of agent actions
CREATE TABLE IF NOT EXISTS events (
    id          BIGSERIAL PRIMARY KEY,
    agent_id    UUID REFERENCES agents(id) ON DELETE CASCADE,
    event_type  VARCHAR(64) NOT NULL,
    payload     JSONB      NOT NULL,
    occurred_at TIMESTAMPTZ DEFAULT now()
);

-- 3️⃣ Potential Loss table – snapshot of computed risk metrics
CREATE TABLE IF NOT EXISTS potential_loss (
    id            BIGSERIAL PRIMARY KEY,
    agent_id      UUID REFERENCES agents(id) ON DELETE SET NULL,
    loss_value    NUMERIC(18,4),
    risk_level    VARCHAR(16),      -- LOW / MEDIUM / HIGH
    computed_at   TIMESTAMPTZ DEFAULT now(),
    source_file   TEXT              -- for traceability
);

-- 4️⃣ Audit Trail – captures source grounding metadata
CREATE TABLE IF NOT EXISTS audit_trail (
    id          BIGSERIAL PRIMARY KEY,
    event_id    BIGINT REFERENCES events(id) ON DELETE CASCADE,
    source      TEXT NOT NULL,          -- e.g., 'external_api', 'file_import'
    details     JSONB,
    created_at  TIMESTAMPTZ DEFAULT now()
);
```

> **왜?**  
> - `agents` → 에이전트별 식별과 역할 부여.  
> - `events` → 모든 외부/내부 동작을 무결성 있게 기록.

## 2️⃣ 초기 데이터 인제스트 구조

```python
# data_ingest.py
"""
⚙️ 데이터 인제스트 유틸리티

- CSV/JSON 파일을 읽어 agents / events 테이블에 삽입
- PostgreSQL 연결은 환경 변수 사용 (PGHOST, PGUSER, PGPASSWORD, PGDATABASE)
"""

import os
import csv
import json
from pathlib import Path
import psycopg2
from psycopg2.extras import execute_values

DB_CONN = {
    "host": os.getenv("PGHOST", "localhost"),
    "port": int(os.getenv("PGPORT", 5432)),
    "dbname": os.getenv("PGDATABASE", "jks_db"),
    "user": os.getenv("PGUSER", "postgres"),
    "password": os.getenv("PGPASSWORD", ""),
}

def get_conn():
    return psycopg2.connect(**DB_CONN)

def ingest_agents(csv_path: str):
    """
    CSV 예시:
    name,role
    Writer,content_creation
    Researcher,data_analysis
    """
    with Path(csv_path).open(newline="", encoding="utf-8") as f:
        reader = csv.DictReader(f)
        rows = [(row["name"], row.get("role")) for row in reader]

    sql = """
        INSERT INTO agents (name, role)
        VALUES %s
        ON CONFLICT (name) DO UPDATE SET role = EXCLUDED.role;
    """
    with get_conn() as conn, conn.cursor() as cur:
        execute_values(cur, sql, rows)
    print(f"[ingest_agents] {len(rows)} agents inserted/updated.")

def ingest_events(json_path: str):
    """
    JSON 예시:
    [
        {
            "agent_name": "Writer",
            "event_type": "content_generated",
            "payload": {"title":"...", "length":1200}
        },
        ...
    ]
    """
    with Path(json_path).open(encoding="utf-8") as f:
        events = json.load(f)

    rows = []
    for ev in events:
        # Resolve agent_id
        with get_conn() as conn, conn.cursor() as cur:
            cur.execute("SELECT id FROM agents WHERE name = %s;", (ev["agent_name"],))
            res = cur.fetchone()
            if not res:
                raise ValueError(f"Agent {ev['agent_name']} not found")
            agent_id = res[0]

        rows.append((agent_id, ev["event_type"], json.dumps(ev["payload"])))

    sql = """
        INSERT INTO events (agent_id, event_type, payload)
        VALUES %s;
    """
    with get_conn() as conn, conn.cursor() as cur:
        execute_values(cur, sql, rows)
    print(f"[ingest_events] {len(rows)} events inserted.")

def main():
    ingest_agents("agents.csv")      # 상대 경로는 프로젝트 루트 기준
    ingest_events("events.json")

if __name__ == "__main__":
    main()
```

> **왜?**  
> - `ingest_agents` → 에이전트 목록을 한 번에 로드, 중복 방지.  
> - `ingest_events` → 이벤트를 기록하면서 agent lookup 수행, 무결성 보장.  
> - `execute_values` → 대량 삽입 성능 최적화.

## 1. 목표
**잠재적 손실액(Potential Loss, PL)**를 계산하여 재무 리스크 최소화와 SaaS 가격·가치 제안에 반영한다.

## 2. 핵심 KPI
| KPI | 정의 | 단위 | 비고 |
|-----|------|------|------|
| **연간 예상 손실액 (Annual PL)** | 과거 거래(과거 30일)에서 발생한 환불·분쟁·오류로 인한 실제 손실액 합계 | USD | 현 데이터 기반 초기 추정 |
| **잠재적 손실 대비 방어비율 (Coverage Ratio)** | `PL / (월 평균 매출)` | % | 방어 비용이 매출 대비 얼마나 적절한지 판단 |
| **평균 손실 발생 빈도 (Loss Frequency)** | `발생 거래수 / 총 거래수` | 횟수/거래 | 리스크 빈도를 파악 |
| **평균 손실 금액 (Avg Loss per Event)** | `PL / 발생 거래수` | USD | 한 번의 리스크가 얼마나 큰지를 측정 |
| **ROI 대비 방어비용 (Return on Guarding)** | `PL / 방어 비용` | 1회당 ROI | 보험료 대비 절감 효과 |
| **잠재적 손실 지표(PL‑Index)** | `PL / (월 평균 매출 * 기간)` | 차트 지표 | 시계열 트렌드 분석 |

## 3. 계산 로직 (초기 프레임워크)

```
# 1. 데이터 수집
transactions = fetch_transactions(lookback_days=30)
refunds      = filter_refunds(transactions)          # 환불, 분쟁 등
loss_events  = identify_loss_events(refunds)

# 2. 손실액 계산
PL_total       = sum(event.amount for event in loss_events)  # USD

# 3. KPI 계산
total_transactions = len(transactions)
loss_frequency     = len(loss_events) / total_transactions

avg_loss_per_event = PL_total / len(loss_events)

monthly_sales      = calculate_monthly_sales(transactions)   # 월 평균 매출
coverage_ratio     = PL_total / monthly_sales

# 4. 방어비용(예시)
guard_cost = estimate_guard_cost(monthly_sales)           # 예: 1% of monthly sales

roi_guard = PL_total / guard_cost
```

### 설명
- **`fetch_transactions`**: PayPal API(샌드박스)로 과거 30일 거래 데이터 수집.  
- **`identify_loss_events`**: 환불, 결제 실패, 분쟁 발생 등 실제 손실을 초래한 이벤트를 필터링.  
- **`calculate_monthly_sales`**: 거래 금액 합계에서 환불 제외 후 월 평균 매출 산출.  
- **`estimate_guard_cost`**: 초기 가정으로 월 매출의 1%를 방어비용(보험료 등)으로 설정.

# Phase 3: 손실액 시각화 백엔드 아키텍처 초안

## 1️⃣ 목표
- **데이터 인제스트 → 통합 → 계산**  
- **Worker Queue(예: Celery + Redis)** 로 비동기 처리
- **REST API** 를 통해 **손실액 요약 & 시각화 데이터** 제공
- **자동 스케줄링** 으로 매일/주간 리포트 생성

## 2️⃣ 핵심 구성 요소
| Component | 역할 | 주요 기술 |
|-----------|------|-----------|
| **API Gateway** | 외부 요청 라우팅, 인증, rate‑limit | FastAPI + OAuth2 |
| **Worker Service** | 비동기 계산, 데이터 파이프라인 | Celery (Redis broker) + Python |
| **Data Store** | 원시/변환 데이터 저장 | PostgreSQL (TimescaleDB) |
| **Cache Layer** | 빈번 조회 캐시 | Redis |
| **Monitoring** | 작업 상태, 지표 모니터링 | Prometheus + Grafana |
| **CI/CD** | 코드 배포, 테스트 자동화 | GitHub Actions |

### 세부 흐름
1. **API 호출**  
   - `/pl/v1/summary` (GET) → 인증 토큰 검증 → 요청 파라미터(날짜 범위 등)

2. **Worker Queue**  
   - API 게이트웨이에서 `Celery` task(`process_pl_summary`) 실행
   - 작업 ID 반환 → 클라이언트는 `GET /pl/v1/summary/{task_id}` 로 상태 확인

3. **Data Ingestion**  
   - `ingest_transactions` job은 외부 API(예: PayPal, Stripe)에서 과거 30일 거래를 가져와 PostgreSQL `transactions` 테이블에 저장
   - 주기적으로 `cron` 혹은 Celery beat 로 실행

4. **PL 계산**  
   - `calculate_pl` 함수는 `transactions`에서 환불/분쟁을 필터링 → `potential_loss_events`
   - 합계, 빈도, 평균 등 KPI 계산
   - 결과는 `pl_summary` 테이블에 저장

5. **Cache**  
   - 1분마다 최신 `pl_summary`를 Redis에 저장 → API 응답 지연 최소화

6. **API Response**  
   - 캐시에서 조회 → JSON 반환 (`total_pl`, `frequency`, `avg_loss`, `risk_level` 등)

## 4️⃣ Worker Queue 상세 설정

| 항목 | 설정 |
|------|------|
| **Broker** | Redis (`redis://localhost:6379/0`) |
| **Result Backend** | PostgreSQL `celery_results` 테이블 |
| **Concurrency** | 4 (CPU 코어 수에 따라 조정) |
| **Retries** | Exponential backoff, max 5 retries |
| **Health Check** | `/celery/health` endpoint |

```sql
-- init.sql
-- ==========================================================
-- Phase 1: db_primary 테이블 정의 및 기본 인덱스
-- ==========================================================
CREATE TABLE IF NOT EXISTS db_primary (
    id          BIGSERIAL PRIMARY KEY,
    source_id   UUID      NOT NULL,              -- 원본 시스템 식별자
    source_type VARCHAR(50) NOT NULL,           -- 예: 'order', 'inventory'
    payload     JSONB     NOT NULL,              -- 원본 데이터
    ingested_at TIMESTAMPTZ DEFAULT now() NOT NULL,
    processed   BOOLEAN   DEFAULT false,         -- 처리 여부
    error_msg   TEXT      NULL
);

-- 인덱스: 빠른 조회를 위한 source_type + ingested_at
CREATE INDEX IF NOT EXISTS idx_db_primary_source_type
    ON db_primary (source_type, ingested_at DESC);

-- 인덱스: 오류가 있는 레코드 빠른 검색
CREATE INDEX IF NOT EXISTS idx_db_primary_error
    ON db_primary (error_msg IS NOT NULL);
```

```python
"""
data_ingest.py

기본 데이터 인제스트 서비스.
- 외부 API 혹은 파일로부터 원시 데이터를 수집해 db_primary에 저장.
- FastAPI 엔드포인트를 통해 동적 인제스트 지원.

import json
import uuid
from datetime import datetime

import asyncpg
import httpx
from fastapi import FastAPI, HTTPException, BackgroundTasks

# 설정 (실제 환경에서는 .env 혹은 Secret Manager 사용)
DB_DSN = "postgresql://user:pass@localhost:5432/jkstory"

# ------------------------------------------------------------------
app = FastAPI(title="JKstory Data Ingest API")

async def _get_pool() -> asyncpg.Pool:
    return await asyncpg.create_pool(dsn=DB_DSN, min_size=1, max_size=5)

# ------------------------------------------------------------------
@app.post("/ingest")
async def ingest(payload: dict, source_type: str = "unknown"):
    """
    외부에서 JSON payload를 받아 db_primary에 저장.
    """
    pool = await _get_pool()
    async with pool.acquire() as conn:
        try:
            await conn.execute(
                """
                INSERT INTO db_primary (source_id, source_type, payload)
                VALUES ($1, $2, $3)
                """,
                str(uuid.uuid4()),
                source_type,
                json.dumps(payload),
            )
        except Exception as exc:
            raise HTTPException(status_code=500, detail=str(exc))
    return {"status": "queued", "source_type": source_type}


# ------------------------------------------------------------------
@app.get("/ingest/status/{record_id}")

    특정 레코드 처리 상태 조회.
    """
    pool = await _get_pool()
    async with pool.acquire() as conn:
        row = await conn.fetchrow(
            """
            SELECT processed, error_msg FROM db_primary
            WHERE id = $1
            """,
            record_id,
        )
    if not row:
        raise HTTPException(status_code=404, detail="record not found")
    return {"processed": row["processed"], "error_msg": row.get("error_msg")}

@app.post("/ingest/file")
async def ingest_file(file: bytes, source_type: str = "unknown"):
    """
    파일 업로드를 통해 JSON 배열을 한 번에 인제스트.
    """
    try:
        data = json.loads(file)
    except Exception as exc:
        raise HTTPException(status_code=400, detail=f"invalid JSON: {exc}")

    pool = await _get_pool()
    async with pool.acquire() as conn:
        for item in data:
            await conn.execute(
                """
                INSERT INTO db_primary (source_id, source_type, payload)
                VALUES ($1, $2, $3)
                """,
                str(uuid.uuid4()),
                source_type,
                json.dumps(item),
            )
    return {"status": "queued", "count": len(data)}

    """

    """
    pool = await _get_pool()
    async with pool.acquire() as conn:
        exists = await conn.fetchval(
            """
            SELECT EXISTS (
                SELECT 1 FROM information_schema.tables

            )

        )
        if not exists:
            raise RuntimeError("db_primary 테이블이 존재하지 않음. init.sql 실행 필요.")
    await pool.close()
```

```python
# services/kpi_updater.py
import psycopg2
from datetime import date

def update_kpis(conn):
    cur = conn.cursor()
    # 연간 수익·비용 합계 계산
    cur.execute("""
        SELECT SUM((payload->>'revenue')::numeric) AS total_rev,
               SUM((payload->>'cost')::numeric) AS total_cost
        FROM db_primary
        WHERE ingested_at >= date_trunc('year', current_date);
    """)
    rev, cost = cur.fetchone()
    if rev is None or cost is None:
        return
    annual_pl = rev - cost
    coverage_ratio = rev / cost if cost else None

    # KPI 업데이트
    cur.execute("""
        UPDATE db_primary
        SET kpi_annual_pl = %s,
            kpi_coverage_ratio = %s
        WHERE ingested_at >= date_trunc('year', current_date);
    """, (annual_pl, coverage_ratio))
    conn.commit()
```
